------------------------------------------------------------
# 核心架构说明（请严格按此实现）
------------------------------------------------------------

1. 每个 Agent 角色 = 一个独立的 JS 启动文件
   例如：
   - slave-ceo.js
   - slave-cto.js
   - slave-cro.js
   - slave-coo.js
   - slave-pm.js
   - slave-qd.js

2. 每个 JS 文件的作用：
   - 启动并调用 Claude CLI
   - 加载自己的 Agent 系统提示词（身份设定）
   - 连接服务端的 WebSocket 接口
   - 接收任务 → 传给 CLI → 输出结果 → 回传 WS

3. 重点：
   - 仍然使用 Claude CLI
   - 仍然加载各自的角色提示词
   - **不使用子进程**
   - **不做内部进程通信**
   - **不做多进程调度**
   每个 JS 都是独立入口、独立运行、独立调用 CLI。

4. 不实现：
   - 不实现多模型供应商
   - 不实现本地子进程管理
   - 不实现主进程控制子进程

5. 每个 JS 独立部署：
   - 可直接 node 运行
   - 可独立丢进 Docker 运行
   - 可分布在不同机器
   - 全部通过服务端 WS 统一交互

6. 并发规则不变：
   - 包工头（CEO）只有在处理【闲聊、问候、普通对话】时，没有并发限制，可以同时处理多条消息。
   - 包工头在处理【正式任务、业务流程、任务分发】时，同一时间仍然只处理一个任务，后续任务自动排队。
   - 桥王、天文台、蛇头、驴仔、忍者神龟：无论任何场景，同一时间只能处理一个任务，必须排队。

一句话总结：
每个角色一个独立 JS 文件，
每个 JS 独立调用 Claude CLI，
独立加载角色提示词，
独立连接 WS，
不使用子进程架构。
